## Seeds

- **Integration-branch git strategy (config + workflow).** Adopt a two-tier branch model as Cadence's one opinionated flow. Tier 1 = an integration branch per milestone (the cycle's reconciliation point; lands to `main` as one PR + tag). Tier 2 = the ephemeral per-plan worktree branches that fork from the integration tip and are reaped after merge. Best fit for the parallel/worktree path: reconciliation churn stays off `main`, and the file-overlap gate keeps those merges clean by construction (with a stop-and-ask safety net).
  - **Grounded findings** (pressure-test 2026-07-16 of `workflows/execute.md` `execute_parallel`, lines ~147-156): worktree branches `cadence/phase-<N>-plan-<k>` already fork from HEAD, merge back into HEAD sequentially (conflict -> stop & ask), and are deleted in step 4. So integration mode is *mostly already supported when HEAD is a milestone branch*, and the worktree tier already cleans itself up.
  - **What that means for the design:** (a) a `worktree_base` knob is unnecessary - worktrees already fork from the current branch, which is the integration branch when you are on one; (b) the load-bearing gap is branch *ownership* - nothing guarantees you are on an integration branch before parallel execution, so on `main` all reconciliation lands on `main` (the "branch pre-existed by luck" problem from the v1.1 cycle); (c) the other real gap is *integration-branch* cleanup after the final PR merge (return-to-base + reap the merged branch), done by hand this cycle - separate from worktree cleanup, which is already handled.
  - **Proposed config (small, philosophy-preserving):** `git.integration_branch: "milestone" | "trunk"` (default milestone); `git.auto_branch: "ask" | "auto" | "off"` (create/switch to the integration branch at cycle start - THE load-bearing knob); `git.on_land_cleanup: "ask" | "auto" | "off"` (post-merge return-to-base + reap merged integration branch). `git.base_branch` stays the landing/guard base (`main`), distinct from the integration branch. Do NOT add a phase-level branch unit (redundant middle tier) and do NOT touch cad-land's no-preselected-default publish rule - this governs how work is organized, never how it is published.
  - **Where it lands:** `references/git.md` (a fifth rail, or an extension of rail 1), `config.schema.json`, `cad-config`, and the cycle-start workflows (`cad-milestone` setting up the next cycle, `cad-plan`). Self-verify CONTRACTS/config-key tests come along for any new keys.
